1. Vue d'ensemble & architecture

CYNA est une plateforme e-commerce de services de cybersécurité (SOC, EDR, XDR, audit de conformité) proposés sous forme d'abonnements mensuels ou annuels. Le projet est découplé en deux applications indépendantes :

ComposantTechnologieRôle
Cyna_frontPHP 8 (WAMP/Apache)Site public + back-office, rendu serveur (pas de framework)
cyna-apiLaravel 13 / PHP 8.4API REST JSON, logique métier, base de données, paiements

Le front ne se connecte plus directement à une base de données : toutes les données transitent par l'API via HTTP/cURL.

Navigateur
    │  HTTP
    ▼
Cyna_front (PHP / WAMP)         http://localhost/Cyna_front/
    │  cURL  (Bearer token Sanctum)
    ▼
API Laravel (Render)           https://laravel-api-1-zb19.onrender.com/api
    │
    ▼
PostgreSQL (Render/Supabase) + Stripe + SMTP

Gestion de session (point important)

Toutes les pages partagent un seul dépôt de sessions (le dossier par défaut de PHP). config/config.php ne redéfinit pas session_save_path afin d'éviter un « split » entre les pages qui démarrent la session avant ou après l'inclusion de la config. Les données de session clés sont : api_token, utilisateur_id, is_admin, cart.

4. Client API & intégration HTTP

La classe ApiClient (includes/api_client.php) centralise tous les appels. Caractéristiques :

  • Construction de l'URL : API_BASE_URL + '/api'
  • Headers : Accept et Content-Type: application/json
  • Bearer token Sanctum ajouté pour les routes protégées ($auth=true)
  • Stockage de la session après login (storeAuthSession)
  • Repli SSL automatique : si cURL échoue sur un certificat (erreurs 35/60/…), nouvelle tentative sans vérification — utile sous WAMP/Windows.
  • Messages d'erreur explicites (HTTP, JSON, validation Laravel).
// Exemples d'utilisation
api_client()->getProducts(['is_featured' => 'true']);
api_client()->login($email, $password);   // stocke api_token + is_admin en session
api_client()->getProfile();                // route protégée (Bearer)
api_client()->createOrder($payload);

6. Authentification (Sanctum)

  1. Le front envoie POST /auth/login avec email + mot de passe.
  2. L'API retourne { data: { user, token } } (token Sanctum).
  3. Le front stocke le token et l'utilisateur en session (api_token, utilisateur_id, is_admin).
  4. Les requêtes protégées envoient Authorization: Bearer {token}.
  5. POST /auth/logout révoque le token côté API.
La protection csrf.php du front concerne uniquement les formulaires PHP (jeton en session). Elle est indépendante de l'API, qui est stateless (token).

8. Déploiement (Render)

render.yaml définit deux services Docker (région Frankfurt, plan free) :

ServiceTypeCommande
cyna-apiwebphp artisan serve (healthcheck /up)
cyna-api-queueworkerphp artisan queue:work (emails)

Build via Dockerfile : PHP 8.4, extensions pdo_pgsql, pgsql, opcache, bcmath, puis composer install --no-dev --optimize-autoloader.

Cold start : sur le plan gratuit, la première requête après inactivité peut prendre 30–60 s (le service se réveille).

10. Installation locale

Backend (API Laravel)

cd cyna-api
composer install
cp .env.example .env
php artisan key:generate
php artisan migrate
php artisan serve            # http://127.0.0.1:8000

Frontend (WAMP)

  1. Placer Cyna_front dans C:\wamp64\www\
  2. Créer Cyna_front/.env avec CYNA_API_URL
  3. Vérifier que l'extension curl est activée dans php.ini
  4. Ouvrir http://localhost/Cyna_front/index.php
Outils de diagnostic fournis : public/diag_api.php (test connexion API) et public/_diag.php (test session/CSRF). À supprimer en production.
