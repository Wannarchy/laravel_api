1. Vue d'ensemble

CYNA est une plateforme de vente de services SaaS de cybersécurité (SOC, EDR, XDR, etc.) composée de deux applications :

ComposantTechnologieRôle
cyna-apiLaravel 13, PHP 8.4, Sanctum, CashierAPI REST, logique métier, BDD PostgreSQL, paiements Stripe
Cyna_frontPHP 7.4+, Bootstrap 5, Stripe.jsSite vitrine, boutique, espace client, back-office admin

Le frontend communique avec l'API via HTTP/JSON (client cURL). Aucun accès direct à la base de données depuis le site public.

2. Site public (frontend)

Pages disponibles

  • Accueil — carrousel dynamique, texte éditorial, catégories, produits mis en avant
  • Catalogue — liste de tous les services, filtre par catégorie
  • Fiche produit — détails, prix mensuel/annuel, disponibilité, produits similaires
  • Recherche avancée — mot-clé, catégorie, fourchette de prix, disponibilité, tri (pertinence, prix, date)
  • Contact — formulaire de contact
  • À propos, CGU, Mentions légales
  • Connexion / Inscription / Mot de passe oublié / Réinitialisation
  • Confirmation e-mail — validation du compte utilisateur

Navigation

  • Barre de navigation avec lien Catalogue, panier, compte, connexion/déconnexion
  • Sélecteur de langue (menu déroulant FR / EN / AR / HE)
  • Barre de recherche depuis l'accueil
  • Accès admin (badge ⚙ Admin) visible pour les comptes administrateurs

3. Espace client connecté

  • Mon compte — consultation et modification du profil (prénom, nom, e-mail)
  • Mes commandes — historique des commandes passées
  • Mes abonnements — liste des abonnements actifs, annulation possible
  • Adresses — gestion des adresses de facturation/livraison (CRUD)
  • Moyens de paiement — cartes enregistrées via Stripe, définition de la carte par défaut, suppression
  • Déconnexion — invalidation du token API et destruction de la session
Session utilisateur : connexion maintenue 7 jours (cookie persistant). Session admin : déconnexion à la fermeture du navigateur.

4. Catalogue & produits

Affichage

  • Liste et grille de produits avec image, catégorie, prix, badge de disponibilité
  • Produits mis en avant sur la page d'accueil (flag is_featured)
  • Filtrage par catégorie dans le catalogue
  • Produits similaires sur la fiche produit (même catégorie)

Disponibilité & stock

  • Colonne stock en base de données pour chaque produit
  • Statuts affichés : Disponible, Rupture de stock, Indisponible
  • Blocage de l'ajout au panier si produit indisponible ou stock = 0
  • Validation côté API au checkout (refus si stock insuffisant)
  • Décrémentation automatique du stock après commande payée
  • Gestion du stock dans l'admin (création / édition produit)

Tarification

  • Prix mensuel et prix annuel par produit
  • Choix du cycle (mensuel / annuel) à l'ajout au panier

5. Panier & commande

  • Panier — stocké en session PHP, un produit par ligne avec cycle choisi
  • Ajout au panier depuis la fiche produit
  • Recalcul du total selon les prix et la disponibilité
  • Codes promo — validation AJAX avant checkout (montant minimum, expiration, utilisations max)
  • Checkout — nom et adresse de facturation, paiement Stripe (Stripe.js Elements)
  • Confirmation — page de confirmation avec numéro de commande
  • Paiement refusé — page d'erreur avec message explicite

6. Paiements & abonnements (Stripe)

  • Intégration Stripe via Laravel Cashier
  • Modèle par abonnement récurrent (mensuel ou annuel) par produit
  • Synchronisation produits/prix Stripe : commande Artisan stripe:sync-products
  • Champs en BDD : stripe_product_id, stripe_price_id_monthly, stripe_price_id_yearly
  • Checkout API : création de session Stripe Checkout ou paiement direct avec PaymentMethod
  • Webhooks Stripe côté API Laravel (/stripe/webhook) pour finaliser les commandes
  • Gestion des abonnements : statut actif, date de prochaine facturation, annulation
  • Enregistrement des 4 derniers chiffres de carte sur la commande
  • Configuration billing publique (/api/billing/config) — clé publique Stripe pour le front

7. Authentification & comptes

API (Laravel Sanctum — tokens Bearer)

  • Inscription avec confirmation e-mail obligatoire
  • Connexion / déconnexion (révocation du token)
  • Mot de passe oublié et réinitialisation par e-mail
  • Renvoi de l'e-mail de vérification
  • Comptes désactivables (est_actif)
  • Rôles : utilisateur standard et administrateur (is_admin)

Frontend

  • Token API stocké en session PHP (api_token)
  • Envoi automatique du header Authorization: Bearer sur chaque requête authentifiée
  • Protection CSRF sur les formulaires POST du frontend
  • Fallback SSL automatique si certificat cURL invalide en local (WAMP)

8. Multilingue & accessibilité

  • 4 langues : Français, English, العربية (AR), עברית (HE)
  • Support RTL pour l'arabe et l'hébreu (dir="rtl")
  • Traductions centralisées dans includes/lang.php
  • Choix de langue persisté en session, sélecteur dans la barre de navigation
  • Libellés traduits : navigation, catalogue, produit, panier, auth, footer

9. Chatbot

  • Assistant virtuel accessible depuis le site (utilisateurs connectés)
  • Envoi de messages via API POST /api/chat
  • Historique de conversation par session (GET /api/chat/history)
  • Journalisation en base (chat_logs) : message utilisateur, réponse bot, session_id, user_id
  • Admin — Conversations Chatbot : consultation des logs, filtres par date et recherche textuelle, regroupement par session

12. Base de données & infrastructure

Tables principales (PostgreSQL / Supabase)

  • utilisateurs — comptes, rôles, confirmation e-mail
  • categories, products (avec stock, IDs Stripe)
  • orders, order_items
  • product_subscriptions — abonnements liés aux commandes
  • promo_codes
  • homepage_slides, homepage_content
  • user_addresses, user_payment_methods
  • chat_logs
  • personal_access_tokens — tokens Sanctum

Déploiement

  • API — Docker sur Render (PHP 8.4, extensions pdo_pgsql, bcmath)
  • Base — Supabase PostgreSQL (pooler IPv4)
  • Frontend — WAMP local (développement) / serveur web PHP (production)
  • Paiements — Stripe (mode test ou live selon .env)
  • Queue worker Laravel pour e-mails et tâches asynchrones

13. Sécurité

  • Authentification API par token Sanctum (Bearer), pas de cookie Sanctum cross-domain
  • Middleware active — blocage des comptes désactivés
  • Middleware admin — routes admin protégées
  • Protection CSRF sur formulaires frontend
  • Rate limiting sur les routes auth (60 req/min)
  • Mots de passe hashés (bcrypt)
  • Cookies session : HttpOnly, SameSite=Lax
  • Validation des entrées côté API (Form Requests Laravel)
  • Vérification e-mail obligatoire à l'inscription
